## 1. プロジェクト概要

### 目的
アイドルグループ運営向けに、デジタル特典券（チェキ券）の販売・QRコード発行・現場もぎり管理を一元化するSaaSプラットフォームを構築する。

### 主な課題と解決方針

| 課題 | 解決方針 |
|------|----------|
| 物理チケットの偽造・紛失リスク | QRトークンによるデジタル一元管理 |
| 現金決済による集計の手間 | Stripe決済による自動集計・履歴管理 |
| 現場でのチケット確認の煩雑さ | スタッフ向けQRスキャンアプリ |
| 通信不安定な会場での運用 | IndexedDBを活用したオフライン対応 |

### 対象ユーザー

| ロール | 利用シーン |
|--------|------------|
| fan（ファン） | チケット購入・QR表示・購入履歴確認 |
| staff（スタッフ） | 会場でのQRスキャン・もぎり処理 |
| admin（運営管理者） | イベント管理・売上確認・ユーザー管理 |

## 6. 認証設計

### 6-1. 認証フロー

```
[ファン会員登録]
  1. メール + パスワード入力
  2. Supabase Auth でユーザー作成
  3. users テーブルに role='fan' でプロフィール挿入
  4. セッション発行 → ファンアプリにリダイレクト

[ログイン]
  1. メール + パスワード入力
  2. Supabase Auth で認証
  3. users テーブルから role 取得
  4. role に応じて対応アプリのホームへリダイレクト
     - fan   → /tickets
     - staff → /staff
     - admin → /admin
```

### 6-2. セッション管理

- Supabase Auth の JWT セッションを使用
- `autoRefreshToken: true` でトークン自動更新
- `persistSession: true` でブラウザリロード後も維持
- セッション有効期限：Supabase デフォルト（1時間）+ 自動更新

### 6-3. ロールベースアクセス制御（RBAC）

| ロール | アクセス可能画面 | 権限 |
|--------|-----------------|------|
| fan | Fan App全画面 | 自分のチケット閲覧・購入のみ |
| staff | Staff App全画面 | QRスキャン・もぎり処理 |
| admin | Admin Dashboard全画面 + Staff App | 全データ閲覧・編集・管理 |

### 6-4. スタッフ・管理者アカウント発行方針

- スタッフ・管理者アカウントはファン側の新規登録からは作成不可
- Admin Dashboardから管理者が手動でロールを付与する
- 初期管理者アカウントはSupabase Dashboardから直接作成

## 7. 決済設計（Stripe）

### 7-1. 決済フロー

```
[フロントエンド]
  1. ファンが「購入する」ボタンをクリック
  2. Supabase Edge Function（create-checkout-session）を呼び出し

[Edge Function: create-checkout-session]
  3. payments テーブルに status='pending' でレコード作成
     （stripe_session_id を冪等性キーとして使用）
  4. Stripe Checkout Session 作成
  5. Session URL をフロントエンドに返却

[フロントエンド]
  6. Stripe Checkout ページへリダイレクト
  7. ファンが決済情報を入力・完了

[Stripe Webhook → Edge Function: stripe-webhook]
  8. checkout.session.completed イベント受信
  9. Webhook 署名を検証（不正リクエスト排除）
  10. payments.status を 'paid' に更新
  11. tickets テーブルにチケット発行（status='active'）
  12. ticket_types.stock_remaining を減算
  （トランザクション処理 — 全て成功か全てロールバック）

[フロントエンド]
  13. 購入完了ページへリダイレクト
  14. ファンのチケット一覧に新チケット表示
```

### 7-2. 二重発行防止

- `payments.stripe_session_id` に UNIQUE 制約
- Webhook 受信時に既に `status='paid'` であれば処理をスキップ
- チケット発行はトランザクション内で実行（部分的な発行を防ぐ）

### 7-3. 決済失敗時の処理

```
payment_intent.payment_failed イベント受信
  → payments.status を 'failed' に更新
  → チケットは発行しない（payments が 'paid' の場合のみ発行）
```

### 7-4. 返金処理

```
[管理者が返金を実行]
  1. Admin Dashboardから対象チケットを選択
  2. Stripe API で返金処理
  3. payments.status を 'refunded' に更新
  4. tickets.status を 'refunded' に更新
```

## 8. QRチケット設計

### 8-1. QRコードのデータ構造

```json
{
  "ticket_id": "uuid-v4",
  "qr_token": "uuid-v4",
  "event_id": "uuid-v4",
  "issued_at": "2026-05-25T10:00:00Z"
}
```

### 8-2. QRコード生成仕様

| 項目 | 仕様 |
|------|------|
| 形式 | JSON文字列をBase64エンコード |
| サイズ | 300×300px |
| 誤り訂正レベル | H（最高レベル：30%破損まで復元可） |
| 発行タイミング | Stripe決済完了後（Webhookトリガー） |
| 有効期限 | イベント当日まで（valid_until に準拠） |

### 8-3. もぎり処理フロー

```
[スタッフがQRをスキャン]
  1. カメラでQRコードを読み取り
  2. JSON をパースして ticket_id・qr_token を抽出
  3. Supabase で ticket_id を検索

[チケット検証]
  ├── status = 'active'  かつ qr_token 一致 → もぎり確認へ
  ├── status = 'used'                        → エラー（使用済み）
  ├── status = 'expired'                     → エラー（期限切れ）
  ├── qr_token 不一致                        → エラー（無効QR）
  └── event_id 不一致                        → エラー（対象外イベント）

[もぎり確認・実行]
  4. スタッフが確認ボタンをタップ
  5. tickets.status を 'used' に更新（楽観的ロック）
  6. tickets.used_at / used_by を記録
  7. scan_logs に記録
  8. 成功画面を表示
```

## 9. オフライン対応設計

### 9-1. 対象機能

オフライン対応するのは **スタッフアプリのQRスキャン・もぎり処理のみ**。

| 機能 | オンライン | オフライン |
|------|-----------|-----------|
| チケット購入 | ✅ | ❌（要ネット） |
| QR表示 | ✅ | ✅（キャッシュ） |
| QRスキャン | ✅ | ✅ |
| もぎり処理 | ✅（即時反映） | ✅（ローカル保存→後で同期） |
| 管理画面 | ✅ | ❌（要ネット） |

### 9-2. IndexedDB 構造

```
データベース名: idol-saas-offline
バージョン: 1

ObjectStore: scan_queue
  キー: id（UUID）
  インデックス: by_sync_status

レコード構造:
  id:               string   ローカル生成UUID
  ticket_id:        string   スキャンしたチケットID
  staff_id:         string   担当スタッフID
  scanned_at:       string   スキャン日時（ISO8601）
  device_id:        string   端末識別子
  sync_status:      string   pending | synced | conflict | error
  created_locally_at: string ローカル保存日時
```

### 9-3. オンライン復帰時の同期処理

```
[window.addEventListener('online') 発火]
  1. IndexedDB から sync_status='pending' のレコードを全件取得
  2. 各レコードに対して：
     a. Supabase で該当チケットの現在状態を確認
     b. status='active' → もぎり処理実行 → sync_status='synced'
     c. status='used' かつ used_by が別スタッフ
        → sync_status='conflict'（管理画面に警告表示）
     d. エラー → sync_status='error'（再試行待ち）
  3. 同期完了後、synced レコードをクリア
  4. 管理者に conflict 件数を通知
```

### 9-4. 複数端末競合への対応方針

- オフライン中は単一端末での二重スキャンは防げるが、**複数端末間のリアルタイム同期は不可**
- 同期時に競合を検知し、管理者に警告を出す（自動解決はしない）
- 同期後、conflict レコードは管理者が手動で確認・処理する

## 10. セキュリティ設計

### 10-1. Row Level Security（RLS）方針

| テーブル | fan | staff | admin |
|---------|-----|-------|-------|
| users | 自分のみ参照・更新 | 参照不可 | 全件参照・更新 |
| events | 公開中のみ参照 | 公開中のみ参照 | 全操作 |
| ticket_types | 参照のみ | 参照のみ | 全操作 |
| tickets | 自分のみ参照 | 全件参照・ステータス更新 | 全操作 |
| payments | 自分のみ参照 | 参照不可 | 全操作 |
| scan_logs | 参照不可 | 挿入・参照 | 全操作 |

### 10-2. シークレット管理

| キー | 管理場所 |
|------|---------|
| Stripe 秘密鍵（sk_live_...） | Supabase Edge Functions の環境変数のみ |
| Supabase サービスロールキー | Edge Functions の環境変数のみ |
| Supabase ANON キー | フロントエンド `.env.local`（公開可） |
| Stripe 公開可能キー | フロントエンド `.env.local`（公開可） |

> **厳守事項：** Stripe 秘密鍵・サービスロールキーは絶対にフロントエンドコードに含めない。

### 10-3. Webhook セキュリティ

- Stripe Webhook の署名検証（`stripe.webhooks.constructEvent`）を必ず実装
- 署名検証失敗時は HTTP 400 を返し処理を中断

### 10-4. その他のセキュリティ対策

- HTTPS 強制（Vercel デフォルト）
- Supabase Auth による JWT 検証
- レート制限（Supabase + Vercel Edge Runtime）
- XSS 対策：React のデフォルトエスケープ + dangerouslySetInnerHTML 不使用

## 11. API方針

### 11-1. 基本方針

本プロジェクトは **専用RESTful APIサーバーを持たない** アーキテクチャを採用する。
データアクセスは **Supabase クライアント（PostgREST）** を通じて直接行う。
サーバーサイド処理が必要な機能（決済・Webhook）のみ **Supabase Edge Functions** を使用する。

### 11-2. Supabase Edge Functions 一覧

| Function名 | トリガー | 処理内容 |
|-----------|---------|---------|
| `create-checkout-session` | フロントエンドから呼び出し | Stripe Checkout Session 生成 |
| `stripe-webhook` | Stripe Webhook | 決済完了後のチケット発行 |

### 11-3. データアクセスパターン

```typescript
// 例：ファンのチケット一覧取得
const { data, error } = await supabase
  .from('tickets')
  .select(`
    id,
    status,
    qr_token,
    purchased_at,
    ticket_type:ticket_types(name, price),
    event:events(title, date, venue)
  `)
  .eq('user_id', user.id)
  .eq('status', 'active')
  .order('purchased_at', { ascending: false })
```

### 11-4. Realtime 活用方針

| 機能 | Realtime 使用 | 理由 |
|------|--------------|------|
| もぎり状態反映 | ✅ | 複数スタッフ間での即時同期 |
| 在庫数表示 | ✅ | リアルタイム残数表示 |
| 管理ダッシュボード | ✅ | 売上・スキャン数のライブ更新 |

## 12. 開発方針・Git運用

### 12-1. ブランチ戦略

```
main
  └── 本番リリースのみ。直接プッシュ禁止。
      PRレビュー必須。

develop
  └── 統合ブランチ。全 feature/* / fix/* の PR ターゲット。
      CI通過 + レビューで main へマージ。

feature/*
  └── 新機能開発ブランチ。
      例: feature/m2-stripe-checkout

fix/*
  └── バグ修正ブランチ。
      例: fix/qr-token-validation
```

### 12-2. コミットメッセージ規約

```
feat:     新機能
fix:      バグ修正
chore:    設定・ビルド関連
docs:     ドキュメント
refactor: リファクタリング
test:     テスト追加・修正

例：
feat: M2 Stripe checkout session API実装
fix: QRトークン検証の二重スキャン防止ロジック修正
```

### 12-3. 環境分離

| 環境 | ブランチ | Supabase | Stripe |
|------|---------|----------|--------|
| 開発 | develop | 開発用プロジェクト | テストキー |
| 本番 | main | 本番プロジェクト | 本番キー |
